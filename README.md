Из заголовков каждого раздела отфильтровываются инструкции по монтажу/эксплуатации/обслуживанию — публикуется только продающий контент (виды и назначение товара, бренды и т.д.), см. `INSTRUCTION_TITLE_RX` в `scraper.js`.

### Режимы работы

**Первый запуск** — если база данных пуста:

- Парсер обходит все разделы и сохраняет найденные статьи в базу как «уже виденные»
- Ни одна статья не публикуется
- После завершения программа переходит в режим мониторинга

**Режим мониторинга** — основной режим:

- Каждые N минут (задаётся в `CHECK_INTERVAL_MINUTES`) парсер проверяет все разделы
- Статьи, которых нет в базе — новые: парсятся полностью и публикуются
- Уже виденные статьи пропускаются

### Публикация

**Telegram:**

- Если у статьи есть изображение — публикуется как фото с подписью
- Если изображения нет или не удалось скачать — публикуется как текстовое сообщение
- Изображение скачивается через браузерный `fetch` (обход hotlink-защиты)

**ВКонтакте:**

- Публикуется текстовый пост от имени группы
- Если настроен `VK_USER_TOKEN` — фото загружается через пользовательский токен и прикрепляется к посту
- Если `VK_USER_TOKEN` не задан — пост публикуется без фото

### Уведомления об ошибках

При сбое публикации (ошибка Telegram или VK) все администраторы бота получают личное сообщение с описанием ошибки.

## Архитектура

```
main.js          — точка входа, cron-расписание, оркестрация
├── scraper.js   — Playwright: парсинг списков и статей
├── formatter.js — форматирование текста для TG и VK
├── publisher.js — отправка в Telegram API и VK API
├── database.js  — JSON-база данных (data/articles.json)
├── bot.js       — Telegram-бот (long polling, кнопки)
├── config.js    — конфигурация из .env
└── logger.js    — Winston-логгер + уведомления админам
```

**База данных** — файл `data/articles.json`. Каждая статья хранит:

- `url`, `title`, `section`, `articleType`, `publishedAt`
- `postedTg` / `postedVk` — флаги публикации
- `tgMsgId` / `vkPostId` — ID опубликованных постов
- `scrapedAt` — время добавления в базу

## Установка на сервер

### Требования

- Ubuntu 20.04+ / Debian 11+
- Node.js 22+
- 1 ГБ RAM минимум (Playwright + Chromium)
- ~500 МБ свободного места (браузер)

### Шаг 1 — Установка Node.js через nvm

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
source ~/.bashrc
nvm install 22
nvm use 22
node --version  # должно показать v22.x.x
```

### Шаг 2 — Клонирование проекта

```bash
mkdir -p /bramy
cd /bramy
git clone https://github.com/DImon1809/bramy_parser.git .
# или распакуй архив src.zip в /bramy/parser/src/
```

Структура должна быть:

```
/bramy/
├── data/          # создаётся автоматически
├── logs/          # создаётся автоматически
└── parser/
    ├── src/
    ├── .env        # создай вручную (см. ниже)
    └── package.json
```

### Шаг 3 — Установка зависимостей

```bash
cd /bramy/parser
npm install
```

### Шаг 4 — Установка браузера Playwright

```bash
npx playwright install chromium --with-deps
```

> Скачивает ~300 МБ. Флаг `--with-deps` устанавливает системные библиотеки Ubuntu.

### Шаг 5 — Создание .env

```bash
cp .env.example .env
nano .env
```

Заполни все переменные (см. раздел [Настройка .env](#настройка-env)).

### Шаг 6 — Установка pm2 и запуск

```bash
npm install pm2 -g
pm2 start src/main.js --name bramy
pm2 save
pm2 startup  # автозапуск при перезагрузке сервера
```

## Настройка .env

Создай файл `/bramy/parser/.env`:

```env
# ── Telegram ──────────────────────────────────────────────
# Токен бота (получить у @BotFather)
TG_BOT_TOKEN=1234567890:AAExxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

# ID канала для публикации (например: @my_channel)
TG_CHANNEL_ID=@bramy_test

# Telegram ID администраторов (через запятую — для нескольких)
TG_ADMIN_ID=663828088,987654321

# ── ВКонтакте ─────────────────────────────────────────────
# Токен группы (Управление группой → Работа с API → Ключи доступа)
# Права: Управление, Стена
VK_TOKEN=vk1.a.xxxxxxxx...

# Пользовательский токен для загрузки фото (см. инструкцию ниже)
# Права: photos, wall, groups
VK_USER_TOKEN=vk1.a.xxxxxxxx...

# ID группы ВКонтакте (число без минуса)
VK_GROUP_ID=237027004

# ── Парсер ────────────────────────────────────────────────
# Интервал проверки в минутах
CHECK_INTERVAL_MINUTES=10
```

## Получение VK токенов

### VK_TOKEN — токен группы

1. Перейди в свою группу ВКонтакте → **Управление**
2. **Работа с API** → **Ключи доступа** → **Создать ключ**
3. Отметь права: **Управление**, **Стена**
4. Скопируй токен → вставь в `VK_TOKEN`

> Групповой токен работает с любого IP и не требует привязки.

### VK_USER_TOKEN — пользовательский токен (для фото)

Нужен для загрузки изображений к постам — методы `photos.getWallUploadServer`/`photos.saveWallPhoto` не работают с групповым токеном (VK API возвращает `error 27: method is unavailable with group auth`), только с личным. Токен должен быть получен **с IP-адреса сервера**, иначе VK заблокирует его.

**Через SSH-прокси:**

1. На локальном компьютере создай SOCKS-тоннель:

   ```
   ssh -D 9090 -N root@АДРЕС_СЕРВЕРА
   ```

   Окно зависнет — это нормально, не закрывай.

2. Запусти браузер через этот прокси:
   - **Firefox**: Настройки → Сеть → Ручная настройка → SOCKS v5, хост `localhost`, порт `9090`
   - **Chrome** (командная строка):
     ```
     "C:\Program Files\Google\Chrome\Application\chrome.exe" --proxy-server="socks5://localhost:9090"
     ```

3. В этом браузере открой ссылку (подставь свой `client_id`):

   ```
   https://oauth.vk.com/authorize?client_id=2685278&scope=wall,photos,groups,offline&response_type=token&redirect_uri=https://oauth.vk.com/blank.html&v=5.199
   ```

   Важно: scope обязательно должен включать `offline` — без него VK выдаёт токен с ограниченным сроком жизни (`expires_in` в ответе), а с `offline` токен постоянный (`expires_in=0`) и не протухает сам по себе, только по ручному отзыву или смене пароля.

4. Авторизуйся → в адресной строке скопируй значение `access_token=...` (там же можно проверить `expires_in=0` — если значение не 0, значит `offline` не применился)

5. Вставь токен в `VK_USER_TOKEN` на сервере, перезапусти бот:

   ```bash
   pm2 restart bramy
   ```

6. Отключи прокси в браузере.

> Токен будет привязан к IP сервера и будет работать корректно.

## Управление через pm2

```bash
# Просмотр статуса
pm2 list

# Логи в реальном времени
pm2 logs bramy --lines 50

# Перезапуск (после изменения кода или .env)
pm2 restart bramy

# Остановка
pm2 stop bramy

# Удаление из pm2
pm2 delete bramy
```

### Обновление кода

```bash
cd /bramy/parser
git pull
pm2 restart bramy
```

### Переключение версии Node.js

pm2 устанавливается под конкретную версию Node. Если после переподключения pm2 не найден:

```bash
source ~/.bashrc
nvm use 22
pm2 restart bramy
```

## Ограничение ресурсов сервера

Сервер слабый (1 vCPU / 1 ГБ RAM), а Chromium под Playwright под нагрузкой может занять почти всё ядро (см. `CHROMIUM_ARGS` и комментарии в `scraper.js` про `--single-process` и переполнение памяти). Ниже — что настроено, чтобы парсер не мог положить сервер целиком.

### Своп-файл

Без свопа при пиковой нагрузке возможен OOM-kill процесса даже с периодическим пересозданием страницы (`PAGE_RECYCLE_EVERY_SECTIONS` в `scraper.js`):

```bash
fallocate -l 2G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
free -h
```

### CPU-квота через systemd

`pm2 startup` только печатает команду для регистрации автозапуска — саму строку (начинается с `sudo env PATH=...`) нужно скопировать из вывода и выполнить отдельно, иначе systemd-юнит не создастся:

```bash
pm2 startup
# скопировать и выполнить строку, которую команда выведет
pm2 save
```

После этого демон pm2 — а вместе с ним node-процесс и все процессы Chromium, они попадают в тот же cgroup — управляется юнитом `pm2-root.service`. Чтобы ограничить суммарное CPU всего этого дерева процессов:

```bash
sudo systemctl edit pm2-root
```

В дроп-ин (`/etc/systemd/system/pm2-root.service.d/override.conf`) добавить:

```ini
[Service]
CPUQuota=70%
```

Применить:

```bash
sudo systemctl daemon-reload
sudo systemctl restart pm2-root
```

Проверить, что квота реально применилась:

```bash
cat /sys/fs/cgroup/system.slice/pm2-root.service/cpu.max
# должно быть "70000 100000" (= 70%), а не "max"
```

`CPUQuota` — это троттлинг, а не убийство процесса: при превышении парсер просто работает медленнее, а не падает. Если в логах (`pm2 logs bramy`) участятся таймауты `Не удалось открыть ...` — квоту можно ослабить (поменять цифру в `override.conf`, затем снова `daemon-reload` + `restart`).

> **Лимит по памяти сознательно не ставился** (ни `--max-memory-restart` в pm2, ни `MemoryMax`/`MemoryHigh` в systemd). Жёсткий рестарт по памяти может оборвать `browser.close()` в `finally`-блоке (`scraper.js`), оставив осиротевшие процессы Chromium, либо прервать цикл публикации на середине — статья к этому моменту уже помечена в базе как «виденная» (`db.save`), но опубликована не будет, а автоматического повторного ретрая для неё нет (кнопка «Тестовый запуск» в боте публикует только самую последнюю статью). Своп + CPUQuota закрывают риск OOM без этого побочного эффекта.

## Telegram-бот

Бот доступен только пользователям, чьи Telegram ID перечислены в `TG_ADMIN_ID`.

### Команды

| Действие                      | Описание                                                                                                                                    |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `/start`                      | Показать приветствие и главное меню                                                                                                         |
| Кнопка **📊 Состояние**       | Статистика: всего статей в базе, опубликовано (TG или VK), последняя проверка                                                               |
| Кнопка **🚀 Тестовый запуск** | Публикует самую свежую статью из базы в TG и VK. Показывает результат по каждой платформе. Если статья уже опубликована — сообщает об этом. |

### Добавление администраторов

В `.env` через запятую:

```env
TG_ADMIN_ID=663828088,123456789,987654321
```

Чтобы узнать свой Telegram ID — напиши боту [@userinfobot](https://t.me/userinfobot).
